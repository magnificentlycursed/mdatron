## Summary

mdatron is a methodology-agnostic conformance engine for typed markdown, designed agent-first. It validates structure (JSON Schema over frontmatter), cross-file and registry semantics (a narrowed rule DSL), and adopter-supplied pattern data (routes, content-hash pins, vocabularies, citations), emitting compiler-shaped diagnostics. Agents are its first consumer on the engine's own terms: any project whose work is executed by agents benefits from action-time conformance feedback, and vsdd is the first adopter rather than the reason. The repo is a two-crate Rust workspace (`mdatron-core` library, `mdatron-cli` binary; the library is consumed in-process by adopting tools, the binary by everyone else). The engine ships `verify` and `explain` today and adds `init` in this increment. The human terminal surface is a view of the agent-facing contract.

Emphasis, inverted from the superseded design (`DESIGN-MDATRON.md`, whose center was the rule DSL and its imagined built-in patterns): the check families are the center of the tool; the DSL is a kept component serving one lane.

## Project declarations

**Surfaces:** cli-input, file-io, untrusted-markdown (parsers are a trust boundary), ai-runtime (diagnostics are consumed by agents in context).
**Review config:** thorough — the engine gates other projects' work; its defects propagate. Mutation floor declared; the floor's number is set in phase 1b with the fixture corpus.
**Phase 5 strategy:** mutation testing on the engine with the kill-ratio floor in CI; fuzz testing on every input parser — frontmatter, DSL pattern files, JSON Schema documents, route tables, pin manifests, vocabulary registries. No proof execution. Pin hashes use sha256; collision resistance is inherited from the algorithm, and "no proof execution" refers to formal proof tooling, not hashing. Pattern matching uses linear-time engines; adopter-supplied regular expressions (JSON Schema `pattern`, vocabulary rules) run under the step budget below.
**Phase 6 strategy:** at the v1.0 milestone close — specification, tests, implementation, and verification each complete and mutually consistent.

## Behavioral contracts

Referenced by exact heading name. Each contract states normative content, then falsification, then evidence.

### Agents are the first consumer

The engine's interfaces are designed for agent consumption first; every other surface is a view of that contract.

- Diagnostics are designed to be acted on by an agent. The rustc shape is a deliberate choice: agents carry deep training priors on compiler errors, so the format is itself a feedback mechanism.
- Diagnostic output has a token cost, and that cost is a design constraint. The compact form carries a fixed per-finding size limit (the number is set in phase 1b alongside the mutation floor; it is a contract limit, not a band derived from actuals); sizes are measured and recorded in the cost ledger (the governing contract's usage record).
- Text that flows into agent context is a trust surface. Engine-authored text is the only unmarked text in a diagnostic. Adopter content — interpolated frontmatter values, flagged prose in vocabulary findings, and governance-annotation text when surfaced — is carried in the JSON envelope as a structurally distinct quoted field, and rendered in the TTY and compact forms as prefix-marked regions: after escaping, every line of quoted content carries the quote prefix. Interpolated values never render inline: the message line carries an engine-authored placeholder, and the value renders as a prefix-marked block beneath it, because an inline marking delimiter would be forgeable by the same argument that rules out closing delimiters. The rendering alphabet is a partition, because a code point cannot be both escaped-to-visible and split-upon: the split set — the line-break code points LF, CR, VT, FF, NEL, and the separator classes Zl and Zp — is consumed as breaks, with every resulting line prefixed; the escape set — the remaining control characters (Cc) — renders as inert visible escapes. The guarantee is two-legged: every code point any consumer may treat as a line break lies in the partition's universe (Cc, Zl, Zp), and the partition sends each such point either to prefixed-break (split set) or to inert visible escape (escape set) — so no raw break byte survives for any consumer, including consumers that split on escape-set members such as FS, GS, and RS. Neither leg alone carries the property: the escape leg is what neutralizes ANSI controls and the exotic splitters, and it is load-bearing. When the size limit forces truncation of quoted content, the cut lands only at a line boundary and an engine-authored elision marker closes the region; no cut occurs mid-line or mid-escape. A prefix scheme has no closing delimiter, so adopter bytes cannot forge an end-of-quote: producing an unprefixed line is impossible when the renderer prefixes every line of the region.
- No invocation path is interactive: no prompt, no confirmation, no TTY dependency. An agent can never hang on a question.
- Output is deterministic: identical input produces byte-identical output in every form — stable finding order, no timestamps or run-specific values, and file paths rendered relative to the governed tree root so output is identical across checkouts.
- No silent degradation: every unverifiable state — bound exceeded, malformed data, unreadable file, a family inactive for missing data — is a reported state. Silence always means verified-clean, never unverified.
- Styling carries no information: the TTY form stripped of styling equals the plain rendering; nothing actionable is expressed only in color or layout.

Falsification: an interactive prompt reachable in any invocation; two runs on identical input differing byte-wise in any output form, including across checkout locations; a compact finding exceeding the size limit; adopter-derived text appearing in a diagnostic unmarked, unescaped, inline in an engine-authored line, or outside a prefix-marked region in a rendered form; a quoted region containing an unprefixed line under the split set; a truncation cutting mid-line or mid-escape, or leaving a quoted region without its elision marker; an unverifiable state producing no report; a stripped TTY rendering differing in content from the plain rendering.

Evidence: diagnostics flow into agent context by design; round 1 verified two injection channels carrying adopter bytes verbatim into messages (message interpolation at verify.rs:428-441; quoted prose in vocabulary findings) with escaping covering only the path field (diagnostic.rs:58-69), and round 2 showed that any closing-delimiter marking scheme is forgeable by adopter bytes while control-character escaping cannot stop printable payloads — the prefix scheme is the response to both. The rustc-shape rationale is recorded in the operator interview.

### Diagnostics are a versioned contract

Machine output is a public interface with version discipline.

- The JSON envelope (`mdatron_output_version`, findings, summary, pipeline status, the quoted-content fields) is an established, published contract: changes to its shape or to the meaning of an emitted code require a version bump. Code ranges are reserved per family; the reserved-range table is enforced in CI.
- Every emitted code has an explain page and a fixture pair; the catalog grows one entry per emitted code, with no seeded-count targets.
- Output forms: rustc-shaped TTY for humans, JSON for machines, compact for agent context. All three render the same findings.

Falsification: an emitted code absent from the catalog; an envelope-shape or code-meaning change without a version bump; a code emitted outside its family's reserved range surviving CI; the three output forms disagreeing on findings for the same input.

Evidence: the reserved-code semantic swap shipped as a silent contract break and was caught by three review lenses independently (vsdd-cli tracker, #12 children).

### Validation is data-driven

The engine knows no methodology. Every check family executes adopter-supplied data.

- Inputs per family: JSON Schemas, rule pattern files, route tables, pin manifests, vocabulary registries, citation conventions. vsdd is the first adopter and supplies its own data; the engine ships none of it. (The superseded design's informal-influence note carries forward: the DSL's coverage scope was informed by the first adopter's review evidence, as a worked example rather than a dependency.)
- The engine validates its own inputs: route tables, pin manifests, and registries have engine-shipped schemas (they are engine-defined interfaces), and malformed adopter data yields a diagnostic, never a panic.
- Conflicting supplied data is a diagnostic, with defined precedence: two routes claiming one file is an error; a term simultaneously registered and draft resolves to draft (the permissive reading) with a warning.
- No methodology vocabulary appears in engine source, engine-authored diagnostic text, or engine-shipped documentation beyond naming vsdd as an adopter. The consume graph is empty: mdatron depends on no methodology artifact, and its dependency manifest is checkable against an allowlist.
- Governance data is governed: `.mdatron/` is itself routed and pinned. The managed partition is defined by the init manifest — files the manifest lists (with content hashes) are managed, and drift in them is refused; adopter-authored data lives outside the manifest. The manifest cannot carry its own hash (a fixed point); its integrity anchor is stated plainly: the manifest's content hash is held by the pin family in a separate governing record, and the terminal anchor is commit review. Removals persist as tombstones: demoting a manifest entry — un-managing a file — replaces the entry with a demotion tombstone recording the removal and its justification (reason and owner), covered like any entry by the manifest's pinned hash. The tombstone is the removal's standing element, and the engine enforces exactly what a single snapshot can show: a naive demotion (manifest edited, pin record stale) trips pin drift, and a tombstoned demotion stays loud through its standing annotation. A competent silent demotion — manifest and pin record rewritten consistently, no tombstone — is invisible to a git-free snapshot by construction; its control is the terminal anchor, commit review, and the design states this plainly rather than asserting an engine signal the engine cannot emit. The reviewable pattern is a manifest entry vanishing without a tombstone: a one-line diff concern that belongs to repository review or CI, outside the engine. Any change to `.mdatron/` inputs invalidates incremental scope and forces whole-tree verification. Changes that weaken governance — additions that permit (a new registry term, a new route, a relaxed schema) and removals that unguard (a tombstoned demotion) — carry an annotated justification that one search enumerates, and the informational lint finding on every whole-tree verify (gates and CI; incremental runs stay scoped to their dependents) fires from the standing annotations themselves: additions carry theirs inline, removals persist through their tombstones, so both shapes stay loud in the tool's own channel for as long as the weakening stands. The owner field is unauthenticated by construction (the engine invokes no git and cannot bind it to the commit author); the operative control is commit review cross-checking the annotation against authorship, and the design says so rather than implying otherwise. Annotation text is adopter content and renders under the marking discipline. The root of trust is the repository's commit review; the engine makes governance weakening loud, not impossible.

Falsification: a check whose behavior requires knowledge not present in supplied data; a methodology term in engine source or engine-authored diagnostic text (checked against an enumerated denylist fixture, scoped to engine-authored strings since quoted adopter content legally appears marked); a dependency from the engine on an adopter artifact; malformed or conflicting adopter data producing a panic or silent misbehavior; a `.mdatron/` change verified under incremental scope; a governance-weakening entry with no justification annotation; an annotated weakening (addition or tombstone) producing no standing informational finding on a whole-tree run; a naive demotion (manifest edited, pin record stale) escaping pin drift; the design or its documentation claiming engine detection for the competent silent demotion.

Evidence: the boundary declaration (`consumes_from: []`, absorbed per the References ledger) held for the entire respec evaluation and the ratified division of labor rests on it; round 2 showed a format-valid but permissive governance change would otherwise keep self-verification clean while disabling checks.

### Five check families

Each family is a generic engine over supplied data, with its own reserved code range and fixture set. Two family-wide disciplines apply. Path confinement: any adopter-supplied path — route entry, pin target, citation, `key()` source — resolves inside the governed tree, with parent-directory, absolute-path, and symlink escapes rejected under the path-confinement code range (`MDATRON-E0010`/`E0011`, carried from the absorbed boundary declarations), including paths whose targets do not exist. Closed-world walking: the engine enumerates the route table's allowlist rather than discovering the tree; symlinks are not followed during enumeration, and unlisted extras are detected by a bounded, no-follow scan — so symlink cycles cannot extend a walk.

- Schema conformance: frontmatter parsing and JSON Schema dispatch per declared artifact class, plus structural checks the schema language cannot express. Implemented today; extended per adopter schemas.
- Route conformance: the route table is an allowlist over the governed tree. A file with no route blocks; a route citing an absent governing document blocks; artifact names underivable from the supplied naming grammar are flagged.
- Pin conformance: governing documents pin a sha256 content hash over the files they govern; a governed-file change with a stale pin fails. Pin recomputation is a single command.
- Vocabulary conformance: prose in governed artifacts is scanned against a supplied registry: unregistered coined terms, invented label schemes (letter-plus-number clusters outside the supplied allowlist), reserved words used outside their registered sense, and listed register anti-patterns are flagged. Draft-status terms are exempt from strict findings.
- Citation conformance: file-and-line citations in findings and governed artifacts are verified against the snapshot of the working tree (the working tree is authoritative per the governing contract as amended; uncommitted edits count; no git subprocess is invoked). A citation naming absent content is rejected.

Falsification, per family: a seeded violation surviving a verify run with that family's data supplied — including a route, pin target, or citation escaping the governed tree by parent-directory, absolute path, or symlink, with existent and non-existent targets both seeded.

Evidence, per family: filename-slug proliferation routed around per-file validation (the review-log slug incident — reviewers invented topic slugs faster than validation could govern; distinct from the review misplacement incident, both in References); governed-file drift caught only by hand sweeps (vsdd-cli #12 children); the letter-cluster pattern recurring past written correction (recurrence records in References); 7 of 8 findings in one review round citing absent code (bookmark-cli-manual, layer 3 round 3); the confinement primitive's non-existent-target hole (the path-confinement defect issue in this tracker) motivates the existent-and-absent seeding.

### Cross-file semantics stay narrowed

The rule DSL serves one lane: cross-file and registry integrity over frontmatter.

- The canonical construct inventory — the complete implemented surface: quantifiers `every`, `some`, `in`; builtins `defined`, `count`, `len`, `union`, `intersect`, `difference`, `concat`, `join`; arithmetic and comparison operators; `let:` bindings; `{{expr}}` message interpolation; the path-confined `key()` cross-file index. Rule patterns are supplied by adopters.
- Body-content extraction is excluded. Expansion requires a falsifiability run over a body-content specification meeting the falsifiability report's 80% bar (the waiver record lives in V1-SHIP-CRITERIA until its conversion, then in the report).
- The engine ships zero built-in patterns; the formerly reserved built-in-pattern code range returns to the pool (the code-ranges open question).

Falsification: documented DSL surface exceeding the inventory above; a seeded body-content construct accepted by the parser or by the evaluator; an expansion landing without the gate.

Evidence: the cold-agent falsifiability run scored 43% against the 80% bar over the full imagined surface (`dsl-falsifiability-report.md`); the narrowed lane is the demonstrated strength.

### Verification is fast where it is invoked

Action-time invocation is the primary performance case, incremental verification is sound, and hook-time cost is bounded. Verification runs on snapshots: each invocation reads its inputs once, through validated no-follow handles with confinement decided on the handle and resolution performed per path component (a symlinked intermediate component is as confined as a symlinked final one), into an immutable snapshot; all checks run against snapshot bytes. The snapshot is what gets verified, the way a compiler compiles the source it read: mutations to the tree after snapshot capture do not affect the in-flight run's results. The no-follow handle read is the Unix implementation (`openat`/`O_NOFOLLOW`): the handle that passed confinement is the handle that is read, so a component swapped after the check cannot redirect the read. On platforms without a handle-relative no-follow open, the fallback performs a per-component symlink check followed by an ordinary open, and a component swapped inside that check-to-open window is followed — a weaker TOCTOU posture the contract tolerates only where the platform offers no `openat` surface, and only until a handle-based walk lands there. This is the one place the snapshot model's check-then-read closure is reopened, per-platform.

- A dependent of a changed file is defined as: any file connected to it through the transitive closure of rule-reference edges (a `key()` or cross-file rule in one file naming the other, in either direction), governance edges (governing document to governed file, in either direction, per the route table), and shared-key edges (two files coupled through the same registry key or keyed rule, without naming each other). The dependent set is computed over all three edge kinds, transitively.
- A hook invocation verifies the changed file plus its dependents, never the whole tree. A change to `.mdatron/` input data is never incremental: it forces whole-tree mode (see Validation is data-driven). Whole-tree verification remains the gate and CI mode. Every run can emit its visited-file set (a trace observable, rendered under the same escaping discipline as diagnostics), so scope claims are testable. The engine's test surface includes a capture-complete seam — an injectable synchronization point marking the moment the snapshot is sealed — so mutation-in-flight fixtures are deterministic rather than racing a window.
- Incremental verification is sound against the whole tree: for any changed file, the incremental result equals the full, unrestricted whole-tree result filtered to findings locatable in the changed file or its true dependents — where the reference dependent sets in fixtures are authored by hand, independent of the engine's own resolution. Neither missing findings nor extra findings are tolerated.
- Hook-time cost is bounded by declared limits shipped as data: maximum input size per file, aggregate snapshot size per invocation, concurrent-invocation count, YAML alias and nesting-depth bounds, extras-scan depth and entry maxima, and a per-invocation step-and-time budget that is global — it covers pattern matching, directory walking, and dependent-closure traversal (shared-key cliques make the closure superlinear, so traversal is inside the budget by design); exceeding a bound is itself a diagnostic (no silent degradation).
- Concurrent invocations are safe: each owns its snapshot, and overlapping runs produce the same results as serial runs. Any future index cache must preserve this property (the index-persistence open question).

Falsification: a hook invocation forced to whole-tree cost; a `.mdatron/` data change verified incrementally; a fixture whose hand-authored dependent set disagrees with the engine's resolution; incremental output missing a finding, or reporting an extra finding, relative to the unrestricted whole-tree result on the changed file and its hand-authored dependents; a visited-file trace naming files outside the computed scope, or rendered outside the escaping discipline; a pathological input exceeding any declared bound without a diagnostic; overlapping invocations producing results that differ from serial runs; an in-flight run whose results reflect a tree mutation made after its snapshot capture, or, on a platform providing handle-relative no-follow open, a confinement check passed on a path that an interleaved swap redirects before the read — the fallback platforms' documented check-to-open window (see the platform carve-out above) is the carved-out exception, not a falsification, until a handle-based walk lands.

Evidence: the governing contract's thesis is feedback latency at action time; round 1 demonstrated the prior agreement oracle was self-referential; round 2 demonstrated the check-then-read gap (confinement checked on a path, the read done on the original path while an agent concurrently edits — the snapshot model closes it on platforms with handle-relative no-follow open; see the platform carve-out for the fallback's tolerated weaker posture), the shared-key edge class that forms no file-to-file reference, and enumeration-time symlink cycles that no per-file bound can reach.

### The engine verifies itself

mdatron's own repository is its first governed tree.

- The repo carries its own `.mdatron/` configuration: schemas for its document classes, a route table over its governed docs, pins, and a vocabulary registry entry set.
- `verify` clean on this repository is a release gate that blocks: a release cannot be cut while self-verification fails. It is also the bootstrap-period exit gate: the exit is recordable only with a clean run's evidence, and the bootstrap validator is deleted per its own criterion at that point.

Falsification: a release proceeding despite failing self-verification (the gate must be shown to block); a bootstrap exit recorded without clean-run evidence (the guard must be shown to refuse).

Evidence: self-verification was discovered absent during cleanup — `verify` on this repo fails with the pipeline-failure diagnostic `MDATRON-E0080`, whose cause here is that no self-configuration exists (cleanup umbrella, this tracker) — leaving the bootstrap validator past its deletion criterion with its gate unsatisfiable.

## Acceptance criteria

The parenthetical names the contract or requirement verified, matching exactly.

- **First-consumer behaviors** (Agents are the first consumer) — no invocation path prompts or blocks on input (verified over the full CLI surface); repeated runs on fixture input are byte-identical in all three forms, including from two checkout locations; compact findings measure within the size limit and the measurements land in the ledger; a seeded adopter-content injection (a frontmatter value carrying instruction-shaped text, a forged quote-prefix line, and values embedding each split-set member — LF, CR, VT, FF, NEL, U+2028, U+2029) appears structurally quoted in the envelope and prefix-marked on every line in the TTY and compact forms; seeded escape-set controls in quoted content (ESC, NUL, and FS — FS standing as class representative, since escape handling is Cc-class membership with no per-code-point branch) render as inert visible escapes with no consumer-splittable raw byte surviving; a seeded over-limit quoted region truncates at a line boundary with its elision marker, including a seed whose naive cut point would land mid-escape; the compact form is asserted as emitted by the verify command itself; a seeded inline interpolation renders as placeholder-plus-quoted-block, never inline; a seeded unverifiable state (unreadable file) is reported; the stripped TTY rendering equals the plain rendering on fixture findings.
- **Family fixtures** (Five check families) — with each family's data supplied, its seeded violation is caught: unrouted file blocks, absent governing doc blocks, underivable name flagged, stale pin fails and passes after re-pin, unregistered coinage and label scheme and reserved-word misuse and listed register anti-pattern flagged with draft terms exempt, dead citation rejected against a dirty working tree (uncommitted content counts), schema break reported, a beyond-schema structural check fires, and a route, pin target, and citation each seeded to escape the governed tree (parent-directory, absolute, symlink; existent and non-existent targets) are rejected; a symlink-cycle fixture terminates with the extras scan bounded.
- **Incremental agreement** (Verification is fast where it is invoked) — on fixture repositories with hand-authored dependent sets: the engine's resolved dependents equal the hand-authored sets (including a three-file transitive chain, the shared-key conflict fixture, a governance-edge fixture exercised in both directions — governed file changed, governing document in the dependents, and the reverse — and a rule-reference fixture likewise exercised in both directions); incremental output equals the unrestricted whole-tree output filtered to the changed file and hand-authored dependents, with no missing and no extra findings; the visited-file trace stays within scope and an unrelated file is absent from it; a `.mdatron/` data change routes to whole-tree mode; a seeded oversized input, a seeded alias-bomb, a seeded depth-bomb, a seeded budget-exhausting pattern, a seeded aggregate-size breach, a seeded concurrent-invocation-count exceedance (N+1 overlapping runs), and a deep-and-wide real-tree fixture (no symlinks) each yield a bound-exceeded diagnostic; the mutation-in-flight fixture mutates the tree at the capture-complete seam and the run reports against its snapshot bytes, including a check-then-swap seed against confinement; overlapping invocations with a mutation between their starts each report against their own snapshots; a seeded adverse filename (control and separator code points in a governed path) renders escaped in the visited-file trace; latency of both modes is measured and recorded (a declared deferral: normal-case thresholds are banded from these actuals in phase 1b, and until then the contract's speed claim rests on the bound diagnostics).
- **Self-verify clean** (The engine verifies itself) — `mdatron verify` runs clean on this repository via its own configuration in CI; a fixture with failing self-verification shows the release gate blocking; a bootstrap-exit attempt without clean-run evidence is refused.
- **Contract stability** (Diagnostics are a versioned contract) — the envelope validates against its published schema; a seeded code-meaning change and a seeded envelope-shape change each fail CI without a version bump; a seeded out-of-range code fails CI; the three output forms agree on fixture findings; every emitted code resolves in `explain` with fixtures.
- **Agnosticism audit** (Validation is data-driven) — the denylist-fixture sweep over engine source and engine-authored diagnostic strings passes; the dependency manifest checks against the allowlist; the engine builds and tests pass with no adopter data present (families inactive, and the inactivity is reported); seeded malformed and seeded conflicting adopter data each yield diagnostics, never panics, and the conflict outcomes are asserted by value and severity (two routes claiming one file yields an error; a registered-and-draft term resolves to draft with a warning); a seeded governance-weakening entry without a justification annotation is flagged; a seeded annotated weakening emits its informational finding on a whole-tree run and does not surface in an unrelated incremental run; the demotion paths are seeded distinctly — the naive path (manifest edited, pin stale) trips pin drift, a tombstoned demotion emits the standing informational finding on subsequent whole-tree runs, and the competent silent path is asserted only as documented commit-review territory with its reviewable pattern stated (no engine signal claimed); and a seeded `.mdatron/` edit under incremental mode is refused to whole-tree.
- **Narrowed surface** (Cross-file semantics stay narrowed) — documentation and `explain` match the canonical construct inventory; a seeded body-content construct is rejected by the parser, and a second seeded construct smuggled past parsing is rejected by the evaluator; the body-content gate is recorded, greppable, and cited by both rejections.
- **Init behaviors** (Init) — first run deploys the skeleton and the init manifest and passes pre-flight; a second run on the unmodified tree is a no-op; a hand-modified managed file causes refusal with a rustc-shaped diagnostic.
- **Sweep completeness** (Citation sweep) — after the sweep, a repo-wide search for the retired filename finds no live reference outside git history and this document's supersession record; a whole-tree verify with citation data reports no dead citations introduced by the retirement.
